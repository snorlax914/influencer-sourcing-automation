## 1. 프로젝트 개요

K-Food에 대한 글로벌 관심이 높아지는 가운데, **한국에 체류·방문 중인 해외 인플루언서**는 현지 팔로워에게 브랜드를 노출할 수 있는 높은 신뢰도의 채널입니다. 그러나 이들을 수작업으로 발굴·평가·컨택하는 과정은 시간과 인력 소모가 크고, 팔로워 수 위주의 주관적 판단에 의존하기 쉽습니다.

본 프로젝트는 **탐색 → 데이터 수집 → 스코어링 → 리스트업 → 개인화 DM 발송 → 회신 분류 → 관계 관리**로 이어지는 전 과정을 n8n 워크플로우와 LLM으로 자동화하여, 비개발 담당자도 오류 없이 지속 운영 가능한 시스템을 만드는 것을 목표로 합니다.

### 해결하려는 문제

| 문제 | 접근 |
|---|---|
| 수동 소싱의 시간·인력 소모 | n8n 파이프라인으로 탐색·수집 자동화 |
| 팔로워 수 위주의 주관적 선별 | 인게이지먼트율·콘텐츠 품질·진성도 기반 정량 스코어링 |
| 내부 데이터의 사일로화 | 협업 이력·성과가 누적되는 통합 CRM DB |
| 개인화 DM 작성의 비용 | LLM 연동 맞춤형 메시지 자동 생성 |
| 회신 처리의 병목 | AI 자동 분류(긍정 / 협의 필요 / 부정) + 후속 액션 라우팅 |

### 기술 스택 (확정 전, 1주차 논의 대상)

| 레이어 | 후보 | 비고 |
|---|---|---|
| 오케스트레이션 | **n8n** (self-hosted / Docker) | 프로젝트 핵심 |
| 데이터 수집 | Instagram Graph API, 서드파티 데이터 제공사(Apify 등) | ⚠️ 아래 [주의사항](#5-데이터-수집-관련-주의사항) 필독 |
| LLM | Claude / GPT API | 콘텐츠 품질 평가, DM 생성, 회신 분류 |
| 데이터베이스 | Supabase(Postgres) / Notion / Google Sheets | 규모 및 팀 역량 고려해 선택 |
| 대시보드 | Looker Studio / Notion / Metabase | 비개발자 운영 관점 우선 |

## 3. 스코어링 모델 (초안)

최종 지표는 2주차에 확정. 현재 설계 방향:

| 지표 | 가중치(안) | 산출 방식 |
|---|---|---|
| 인게이지먼트율 | 30% | (평균 좋아요 + 댓글) / 팔로워 수 |
| 팔로워 진성도 | 25% | 팔로워/팔로잉 비율, 댓글 품질, 급증 패턴 이상치 |
| 콘텐츠 품질 | 20% | LLM 기반 평가 (비주얼 일관성, 캡션 성의, 브랜드 적합도) |
| 게시 빈도·최신성 | 15% | 최근 30일 게시물 수, 마지막 게시일 |
| 한국 체류 신호 | 10% | 위치 태그, 캡션 언어, 최근 게시물 지역성 |

**카테고리 분류 축**
- 국적 / 주요 팔로워 지역
- 사이즈: 나노(~10K) / 마이크로(10K~100K) / 매크로(100K~)
- 콘텐츠 주제: 푸드, 트래블, 라이프스타일, 브이로그 등

## 5. 데이터 수집 관련 주의사항

> **팀 전체가 1주차에 반드시 읽고 합의할 것.**

인스타그램은 **비인가 스크래핑과 자동 DM 발송을 이용약관으로 금지**하며, 계정 정지·법적 리스크가 실제로 발생합니다. 이 프로젝트의 최대 기술적 리스크 지점이므로, 1주차에 아래 옵션 중 무엇으로 갈지 회사 멘토와 명시적으로 합의하고 문서에 남깁니다.

| 옵션 | 장점 | 단점 |
|---|---|---|
| **Instagram Graph API** (공식) | 안전, 지속 가능 | 타 계정 탐색 기능 제한적, 비즈니스 계정 필요 |
| **서드파티 데이터 제공사** (Apify 등) | 탐색 범위 넓음, 법적 책임 분산 | 비용 발생, 데이터 품질 편차 |
| **DM은 반자동(Human-in-the-loop)** | 계정 리스크 최소, 품질 확보 | 완전 자동화는 아님 |

**권장 설계**: 수집은 공식 API + 서드파티 제공사, **DM은 LLM이 초안 생성 → 담당자가 검토·승인 → 발송**하는 반자동 구조. "완전 자동 DM 봇"은 계정이 죽으면 8주 결과물 전체가 무의미해지므로, 데모 안정성 측면에서도 반자동이 유리합니다.

또한 인플루언서 데이터는 **개인정보**입니다. 수집 항목 최소화, 보관 기간 설정, 접근 권한 관리 원칙을 CRM 설계 시 반영합니다.

## 6. 시작하기

### 사전 요구사항
- Docker & Docker Compose
- LLM API 키
- (선택) 데이터 제공사 API 키

### 로컬 실행

```bash
git clone https://github.com/<org>/hallyu-scout.git
cd hallyu-scout

cp .env.example .env
# .env 파일에 API 키 입력

docker compose up -d
```

n8n 접속: http://localhost:5678

### 워크플로우 임포트

```bash
# workflows/ 디렉토리의 JSON 파일을 n8n UI에서 Import
```

> ⚠️ `.env`, 크리덴셜, 수집된 인플루언서 데이터는 **절대 커밋 금지**. `.gitignore` 확인 필수.
